# System Ontology

## Core Purpose

Extracting and validating architectural decisions and domain invariants from code.

## Mental Model

The system operates on an 'agent-first documentation' model, where architectural intent, domain invariants, module boundaries, critical workflows, and cognitive risks are explicitly structured for AI agents. This documentation serves as a stable reference model and a structured checklist for human reviewers during code review, improving alignment and reducing ambiguity.

## Central Concepts

*   **SemanticContext**: The comprehensive, evolving single source of truth for the system's understanding of a codebase, aggregating all derived knowledge. [CLAIM:hyp-4]
*   **Hypothesis**: A provisional statement generated by the system about the codebase, requiring validation and refinement. [CLAIM:hyp-7]
*   **InterviewQuestion**: A structured query generated to validate or refine a Hypothesis, typically requiring human input. [CLAIM:hyp-7]
*   **InterviewAnswer**: The human response to an InterviewQuestion, providing feedback for Hypothesis refinement. [CLAIM:hyp-7]
*   **InterviewSession**: A collection of InterviewQuestions and InterviewAnswers, representing a validation workflow. [CLAIM:hyp-7]
*   **SupportedLanguage**: An enumerated type defining the programming languages the system is designed to recognize and analyze. [CLAIM:hyp-13]
*   **SupportedFramework**: An enumerated type defining the software frameworks the system is designed to recognize and analyze. [CLAIM:hyp-13]
*   **ScanResult**: The output of the initial codebase scanning phase, including identified languages, frameworks, and file metadata. [CLAIM:hyp-13]
*   **LLMConfig**: Configuration parameters specifically for Large Language Model interactions, including provider, model, and operational settings. [CLAIM:hyp-5]
*   **defaultConfig**: The baseline configuration settings for the entire system, including operational parameters and ignore rules. [CLAIM:hyp-5], [CLAIM:hyp-10]

## System Orientation

The system's orientation is **hybrid**: It processes codebase 'state' through a pipeline of 'command'-driven stages (scanning, analyzing, hypothesizing, interviewing, consolidating, generating). It also reacts to external 'events' such as LLM API responses and errors, and is primarily interacted with via 'commands' through its CLI.

## Foundational Principles

*   **Architectural Compliance**: Ensured through explicit agent-first documentation and human review processes, with future CI-based validation strategies planned.
*   **Modularity and Pipeline-Driven Architecture**: The system is built with distinct, interchangeable components, where each stage of the 'AI Context Engineering' process is encapsulated. [CLAIM:hyp-1]
*   **Robust Error Handling and Resilience**: Prioritizes robust error handling for external LLM API calls, distinguishing between recoverable and non-recoverable errors to enable retry mechanisms or graceful degradation.
*   **Single Source of Truth**: The `SemanticContext` serves as the comprehensive, evolving understanding of the codebase, ensuring all derived knowledge is consistently aggregated and versioned. [CLAIM:hyp-4]
*   **High Configurability**: Allows users to customize LLM providers, models, and operational parameters, with a clear hierarchy for configuration resolution. [CLAIM:hyp-5]
*   **Cost and Performance Management**: Explicitly manages LLM token usage as a critical operational constraint to control costs and ensure predictable performance.
*   **Validation and Refinement**: Generated hypotheses are provisional and require mechanisms for validation and refinement, likely through human interaction or further automated analysis. [CLAIM:hyp-7]
*   **Strict Scope Filtering**: Implements a strict policy for ignoring specific directories and file patterns during repository scanning to optimize performance and focus on relevant codebase context. [CLAIM:hyp-10]
*   **CLI-First Design**: All primary interactions and workflows are exposed through command-line commands, rather than a GUI or API server. [CLAIM:hyp-11]

## What This System Is NOT

*   **A file/folder/endpoints/imports/classes inventory**: The system focuses on semantic understanding, not a mere enumeration of code artifacts.
*   **A framework or architecture structure summary**: While it analyzes architecture, its output is not a summary of its own internal framework or architecture.
*   **A Graphical User Interface (GUI) application**: It is designed as a CLI-first application. [CLAIM:hyp-11]
*   **An API server**: Its primary interaction model is through command-line commands, not via HTTP API endpoints. [CLAIM:hyp-11]
*   **A monolithic system**: It explicitly aims for a highly modular, pipeline-driven architecture. [CLAIM:hyp-1]
*   **A system that assumes infallibility of AI-generated outputs**: It implicitly acknowledges that its generated hypotheses are provisional and require validation and refinement. [CLAIM:hyp-7]
*   **A system that analyzes all files indiscriminately**: It has a strict policy for ignoring specific directories and file patterns during repository scanning. [CLAIM:hyp-10]
*   **A system that operates without explicit cost or performance management for LLMs**: It explicitly manages LLM token usage as a critical operational constraint.
*   **A system that supports an unlimited or undefined set of languages/frameworks**: It explicitly supports a predefined set of `SupportedLanguage` and `SupportedFramework` types. [CLAIM:hyp-13]

### Claim Details

**Claim**: The project aims for a highly modular, pipeline-driven architecture, where each stage of the 'AI Context Engineering' process (scanning, analyzing, hypothesizing, interviewing, consolidating, generating) is encapsulated in distinct, interchangeable components within the `core` module. [CLAIM:hyp-1]
**Confidence**: confirmed
**Agent Implication**: Agents should understand the pipeline stages and component responsibilities within `core` to correctly identify where specific logic resides or needs modification, ensuring changes align with the modular design.

**Claim**: The system maintains a comprehensive, evolving 'Semantic Context' as a single source of truth for its understanding of a codebase, ensuring that all derived knowledge (signals, hypotheses, interview data, consolidated knowledge) is consistently aggregated and versioned. [CLAIM:hyp-4]
**Confidence**: confirmed
**Agent Implication**: Agents must always refer to and update the `SemanticContext` as the authoritative source for codebase understanding. Any new insights or modifications to existing knowledge should be integrated into this central structure to maintain consistency and versioning.

**Claim**: The project is designed to be highly configurable, allowing users to customize LLM providers, models, and operational parameters, with a clear hierarchy for configuration resolution (e.g., environment variables overriding config files). [CLAIM:hyp-5]
**Confidence**: inferred
**Agent Implication**: Agents should prioritize using the established configuration mechanisms (config files, environment variables) for LLM and operational parameters. When modifying configuration logic, adhere to the existing resolution hierarchy to avoid unexpected behavior.

**Claim**: The system implicitly acknowledges that its generated hypotheses are not infallible and requires a mechanism for validation and refinement, likely through human interaction or further automated analysis. [CLAIM:hyp-7]
**Confidence**: confirmed
**Agent Implication**: Agents generating or consuming hypotheses should be aware of their provisional nature. When developing new features related to hypothesis generation or refinement, ensure integration with the existing interview and validation mechanisms to facilitate human feedback and status updates.

**Claim**: The system has a strict policy for ignoring specific directories and file patterns during repository scanning to prevent analysis of irrelevant, temporary, or sensitive files, thereby optimizing performance and focusing on relevant codebase context. [CLAIM:hyp-10]
**Confidence**: confirmed
**Agent Implication**: Agents performing file system operations or repository scanning must adhere to the defined `ignoreDirs` and `ignoreFilePatterns` from `defaultConfig`. When adding new file types or directories, evaluate if they should be added to the ignore list to maintain scan efficiency and relevance.

**Claim**: The project is designed as a CLI-first application, where all primary interactions and workflows are exposed through command-line commands, rather than a GUI or API server. [CLAIM:hyp-11]
**Confidence**: confirmed
**Agent Implication**: Agents should assume a CLI-driven interaction model. New features or workflows should be exposed as command-line commands via `src/cli/index.js` and the `commander` library, rather than designing for a GUI or API endpoint.

**Claim**: The system must accurately identify and categorize different programming languages and frameworks within a repository to provide context-aware analysis, and it explicitly supports a predefined set of these. [CLAIM:hyp-13]
**Confidence**: inferred
**Agent Implication**: Agents performing codebase analysis or generating context must rely on the `SupportedLanguage` and `SupportedFramework` types for classification. When extending support for new languages or frameworks, these types must be updated, and detection logic must be implemented accordingly.
